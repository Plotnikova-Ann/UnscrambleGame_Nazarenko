# Unscramble Game

Игра-головоломка для Android, где нужно угадать исходное слово из перемешанных букв.

## Краткое описание

Приложение представляет собой игру, в которой пользователю показывается слово с перемешанными буквами. Задача игрока — ввести правильный вариант исходного слова. За каждый правильный ответ начисляются очки. Игра состоит из 10 слов.

## Технологии

- Kotlin
- Jetpack Compose
- ViewModel
- StateFlow
- Material Design 3
- JUnit (тестирование)

## Функциональность

- Отображение перемешанного слова
- Поле для ввода ответа
- Проверка правильности ответа (без учёта регистра)
- Начисление очков (+20 за правильный ответ)
- Счётчик вопросов (всего 10 слов)
- Кнопка "Пропустить" (переход к следующему слову без начисления очков)
- Сохранение состояния при повороте экрана (ViewModel)
- Диалог завершения игры с итоговым счётом
- Возможность начать игру заново

## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?

**ViewModel** — это компонент Android Jetpack, который хранит данные UI и управляет логикой приложения.

**Какую проблему решает ViewModel:**
- При повороте экрана Activity пересоздаётся, и все данные теряются
- ViewModel сохраняется при configuration changes (поворот, смена темы и т.д.)
- ViewModel живёт дольше, чем Activity

**Что происходит с данными при повороте экрана без ViewModel:**
- Все данные (счёт, текущее слово, прогресс) сбрасываются
- Пользователь теряет прогресс
- Приходится использовать `rememberSaveable` или `onSaveInstanceState()`, что приводит к громоздкому коду

---

### 2. В чём разница между MutableStateFlow и StateFlow?

**MutableStateFlow**  Изменяемый поток данных. Можно обновлять значение через `update()` или `.value` |
**StateFlow** Неизменяемый поток данных. Только для чтения |

**Зачем создавать два поля: _uiState и uiState?**

Это паттерн **"backing property"** (резервное свойство):
- `_uiState` (приватное) — изменяемое, используется только внутри ViewModel
- `uiState` (публичное) — неизменяемое, доступно для чтения из UI

**Что это даёт:**
- Защита данных от случайного изменения из UI
- Инкапсуляция — UI может только наблюдать за изменениями
- Безопасность — состояние меняется только через методы ViewModel

---

### 3. Почему в ViewModel нельзя хранить Context или View?

**Причина:** Утечка памяти (Memory Leak)

**Что может произойти:**
- ViewModel живёт дольше, чем Activity (переживает поворот экрана)
- Если сохранить ссылку на Activity (Context или View), сборщик мусора не сможет её очистить
- Activity остаётся в памяти, даже когда не нужна
- Это приводит к нехватке памяти и падению приложения

**Как правильно:**
- Использовать `AndroidViewModel` если нужен Application Context
- Не передавать в ViewModel Activity, Views, Fragments

---

**Когда лучше использовать каждый подход:**

- **remember** — для простого локального UI (анимация, видимость элементов)
- **rememberSaveable** — для простых данных, которые нужно сохранить при повороте
- **StateFlow + ViewModel** — для бизнес-логики, работы с репозиториями, сложного состояния

### 4. Что такое data class в Kotlin и зачем он используется для UI State?

**data class** — специальный тип класса в Kotlin, который автоматически генерирует полезные методы.

**Какие методы автоматически генерируются:**
- `equals()` — сравнение объектов
- `hashCode()` — вычисление хэш-кода
- `toString()` — строковое представление
- `copy()` — создание копии с изменением отдельных полей
- `componentN()` — деструктуризация

**Зачем нужна функция copy():**
- Позволяет создать новый объект, изменив только нужные поля
- Сохраняет неизменяемость (immutability) данных
- Пример:
```kotlin
_uiState.update { currentState ->
    currentState.copy(score = updatedScore) // только score меняется
}
